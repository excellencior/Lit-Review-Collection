# Mixture of Experts (MoE) — From Overview to Every Detail

> **One-line summary:** Instead of using ALL parameters for every input, MoE routes each input to a small subset of specialized "expert" sub-networks — achieving the knowledge of a giant model at the cost of a small one.

## 1. The Big Picture — Why MoE Exists

### The Scaling Dilemma

There's a well-known law in AI: **more parameters = better performance**. But there's a catch:

```
MORE PARAMETERS = BETTER ... BUT ALSO:

  Parameters    Performance    Cost to Run     Fits on GPU?
  ─────────    ───────────    ───────────     ────────────
    1B          ████           $              ✅ Yes (4GB)
    7B          ████████       $$$            ✅ Yes (16GB)
    70B         ██████████████ $$$$$$$$       ⚠️ Barely (140GB)
    175B        ████████████████████          ❌ No (350GB)
    1T          ██████████████████████████    ❌ Impossible

  Problem: We WANT the intelligence of 175B,
           but we can only AFFORD to run 7B.
```

### The Key Insight

```
  In a 175B dense model processing the word "quantum":

  ┌─────────────────────────────────────────────────────────┐
  │                175 BILLION PARAMETERS                    │
  │                                                         │
  │  Physics neurons:      ████  ← RELEVANT to "quantum"   │
  │  Grammar neurons:      ████  ← somewhat relevant       │
  │  Poetry neurons:       ████  ← NOT relevant            │
  │  Cooking neurons:      ████  ← NOT relevant            │
  │  Legal neurons:        ████  ← NOT relevant            │
  │  Music neurons:        ████  ← NOT relevant            │
  │  Sports neurons:       ████  ← NOT relevant            │
  │  ... 90% of neurons:   ████████████████████████████████ │
  │                        ↑ WASTED COMPUTE!                │
  │                                                         │
  │  ALL 175B parameters are activated for EVERY token,     │
  │  even though most are irrelevant to the current input.  │
  └─────────────────────────────────────────────────────────┘

  What if we could activate ONLY the relevant neurons?
  → That's exactly what MoE does.
```

## 3. Architecture Overview

### Where MoE Fits in a Transformer

MoE **replaces the FFN (Feed-Forward Network)** inside each Transformer layer. The attention mechanism stays exactly the same.

```
STANDARD TRANSFORMER LAYER:

  Input
    │
    ▼
  ┌──────────────┐
  │  Self-        │
  │  Attention    │  ← Same in both dense and MoE
  │  (Q, K, V)   │
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │              │
  │   FFN        │  ← This is what gets replaced!
  │  (2 linear   │     Dense model: ONE large FFN
  │   layers)    │     MoE model:   MANY small FFN "experts"
  │              │
  └──────┬───────┘
         │
         ▼
  Output


MoE TRANSFORMER LAYER:

  Input
    │
    ▼
  ┌──────────────┐
  │  Self-        │
  │  Attention    │  ← Identical to dense model
  │  (Q, K, V)   │
  └──────┬───────┘
         │
         ▼
  ┌──────────────────────────────────────────────────────┐
  │                  MoE LAYER                           │
  │                                                      │
  │   Input ──→ [Router] ──→ picks top-k experts         │
  │                │                                     │
  │     ┌──────┬──┴───┬──────┬──────┬──────┬──────┐     │
  │     │ E₁   │ E₂   │ E₃   │ E₄   │ ...  │ Eₙ   │     │
  │     │(FFN) │(FFN) │(FFN) │(FFN) │      │(FFN) │     │
  │     │ 💤   │ 🔥   │ 💤   │ 💤   │      │ 🔥   │     │
  │     └──────┴──┬───┴──────┴──────┴──────┴──┬───┘     │
  │               │                            │         │
  │               └──────── weighted sum ──────┘         │
  │                           │                          │
  └───────────────────────────┼──────────────────────────┘
                              │
                              ▼
                           Output
```

### Size Comparison

```
┌──────────────────────────────────────────────────────────────┐
│                    DENSE FFN                                 │
│                                                              │
│  Input (d=4096) → Linear (4096 × 16384) → Activation        │
│                 → Linear (16384 × 4096) → Output             │
│                                                              │
│  Parameters: 4096×16384 + 16384×4096 = 134,217,728 (~134M)  │
│  Active per token: 134M (100%)                               │
│                                                              │
╠══════════════════════════════════════════════════════════════╣
│                    MoE FFN (8 experts, top-2)                │
│                                                              │
│  Each expert:                                                │
│    Input (d=4096) → Linear (4096 × 4096) → Activation       │
│                   → Linear (4096 × 4096) → Output            │
│    Parameters per expert: 4096×4096×2 = 33,554,432 (~34M)   │
│                                                              │
│  Router: Linear (4096 × 8) = 32,768 params                  │
│                                                              │
│  Total 8 experts: 8 × 34M = 268M                            │
│  Active per token (top-2): 2 × 34M = 68M  (25%)             │
│                                                              │
│  2× total params, but HALF the compute per token!            │
│  And 2× more total knowledge stored!                         │
└──────────────────────────────────────────────────────────────┘
```

## 4. The Router (Gating Network)

The router is a **small linear layer** that decides which experts should process each token. It's the **most critical and most fragile** component of MoE.

### How It Works

```
Step 1: Compute raw scores for each expert

  Token hidden state: h = [0.3, -0.5, 0.8, 0.1, ...]   (d_model dimensional)
  
  Router weight matrix: W_r  (d_model × num_experts)
  
  scores = h · W_r   →  [2.1, -0.3, 1.7, 0.5, -1.2, 3.4, 0.8, 1.1]
                          E₁    E₂    E₃   E₄   E₅    E₆   E₇   E₈


Step 2: Apply softmax to get probabilities

  probs = softmax(scores)
        = [0.15, 0.01, 0.10, 0.03, 0.01, 0.55, 0.04, 0.06]
           E₁    E₂    E₃   E₄   E₅    E₆   E₇   E₈
                                          ↑
                                     highest prob


Step 3: Select top-k (here k=2)

  Selected: E₆ (prob=0.55) and E₁ (prob=0.15)
  
  
Step 4: Renormalize the selected probabilities

  gate_weights = softmax([0.55, 0.15])
               = [0.60, 0.40]    ← these don't need to match
                                    the original probs, just 
                                    the RELATIVE ranking
```

### Visual: Router Decision for Different Tokens

```
Token: "quantum"
  Router scores:  Physics ████████████  0.55
                  Math    █████         0.20
                  Grammar ██            0.08
                  Poetry  █             0.03
                  Music   ▏             0.01    
  Selected: Physics + Math ✅

Token: "the"
  Router scores:  Grammar ██████████   0.45
                  Syntax  ████████     0.30
                  Physics ██           0.05
                  Poetry  ██           0.05
                  Music   █            0.02
  Selected: Grammar + Syntax ✅

Token: "melancholy"
  Router scores:  Poetry  █████████    0.40
                  Emotion ████████     0.35
                  Grammar ███          0.10
                  Physics █            0.02
                  Music   █            0.02
  Selected: Poetry + Emotion ✅
```

**Key insight:** The router is trained end-to-end via backpropagation. It **learns** which experts are good at what — nobody manually assigns "physics" or "grammar" to experts. The specialization **emerges** through training.

## 5. Expert Networks — The Workers

Each expert is a standard **Feed-Forward Network (FFN)** — two linear layers with a non-linear activation in between:

```
Expert_i(x):

  x    (d_model)
  │
  ▼
  Linear₁: x × W₁ + b₁    (d_model → d_ff)     "expand"
  │
  ▼
  Activation: SiLU / GELU / SwiGLU               "non-linearity"
  │
  ▼
  Linear₂: × W₂ + b₂       (d_ff → d_model)     "compress back"
  │
  ▼
  output   (d_model)
```

### What Makes Experts Different From Each Other?

```
At initialization:
  All experts start with RANDOM (different) weights.
  They are structurally identical but numerically different.

During training:
  Expert 1 sees mostly grammar tokens    → learns grammar patterns
  Expert 2 sees mostly science tokens    → learns science patterns
  Expert 3 sees mostly code tokens       → learns code patterns
  ...

After training:
  ┌──────────────────────────────────────────────────────────┐
  │  Expert 1: W₁ has learned grammar-detecting features     │
  │  Expert 2: W₁ has learned science-detecting features     │
  │  Expert 3: W₁ has learned code-detecting features        │
  │                                                          │
  │  They started the same (structurally) but diverged       │
  │  through training because the ROUTER sent them           │
  │  different types of tokens.                              │
  │                                                          │
  │  The router and experts CO-EVOLVE:                       │
  │  Router learns "Expert 3 is good at code"                │
  │  Expert 3 gets better at code (because it sees more)     │
  │  Router sends even more code to Expert 3                 │
  │  → Self-reinforcing specialization loop! 🔄              │
  └──────────────────────────────────────────────────────────┘
```

## 6. Full Forward Pass — Worked Example With Numbers

Let's trace a single token through a complete MoE layer with **4 experts, top-2 routing**.

### Setup

```
d_model = 4  (tiny for visualization)
num_experts = 4
top_k = 2

Token hidden state (after attention):
  h = [0.5, -0.3, 0.8, 0.2]
```

### Step 1: Router Computes Scores

```
Router weights W_r (4 × 4):
  ┌                          ┐
  │  0.3  -0.1   0.5   0.2  │
  │ -0.4   0.6   0.1  -0.3  │
  │  0.7   0.2  -0.2   0.8  │
  │  0.1  -0.5   0.3   0.4  │
  └                          ┘

scores = h · W_r = [0.5, -0.3, 0.8, 0.2] · W_r

  For Expert 1: 0.5×0.3 + (-0.3)×(-0.4) + 0.8×0.7 + 0.2×0.1
              = 0.15 + 0.12 + 0.56 + 0.02 = 0.85

  For Expert 2: 0.5×(-0.1) + (-0.3)×0.6 + 0.8×0.2 + 0.2×(-0.5)
              = -0.05 - 0.18 + 0.16 - 0.10 = -0.17

  For Expert 3: 0.5×0.5 + (-0.3)×0.1 + 0.8×(-0.2) + 0.2×0.3
              = 0.25 - 0.03 - 0.16 + 0.06 = 0.12

  For Expert 4: 0.5×0.2 + (-0.3)×(-0.3) + 0.8×0.8 + 0.2×0.4
              = 0.10 + 0.09 + 0.64 + 0.08 = 0.91

scores = [0.85, -0.17, 0.12, 0.91]
          E₁     E₂     E₃    E₄
```

### Step 2: Select Top-2

```
scores = [0.85, -0.17, 0.12, 0.91]
               ↓                ↓
          2nd highest       1st highest

Selected: Expert 4 (0.91) and Expert 1 (0.85)
```

### Step 3: Compute Gate Weights (Renormalize)

```
gate_weights = softmax([0.91, 0.85])
             = [exp(0.91)/(exp(0.91)+exp(0.85)), exp(0.85)/(exp(0.91)+exp(0.85))]
             = [2.484/4.824, 2.340/4.824]
             = [0.515, 0.485]

  Expert 4 gets weight 0.515
  Expert 1 gets weight 0.485
```

### Step 4: Run Selected Experts

```
Expert 1 processes h:
  E₁(h) = FFN₁([0.5, -0.3, 0.8, 0.2])
         = [0.42, 0.15, -0.31, 0.67]    (after 2 linear layers + activation)

Expert 4 processes h:
  E₄(h) = FFN₄([0.5, -0.3, 0.8, 0.2])
         = [0.18, 0.73, 0.52, -0.14]    (different weights → different output!)

Experts 2 and 3: NOT activated (compute saved! 💰)
```

### Step 5: Weighted Combination

```
output = gate_weight₄ × E₄(h) + gate_weight₁ × E₁(h)
       = 0.515 × [0.18, 0.73, 0.52, -0.14] + 0.485 × [0.42, 0.15, -0.31, 0.67]

  dim 1: 0.515×0.18 + 0.485×0.42 = 0.093 + 0.204 = 0.297
  dim 2: 0.515×0.73 + 0.485×0.15 = 0.376 + 0.073 = 0.449
  dim 3: 0.515×0.52 + 0.485×(-0.31) = 0.268 - 0.150 = 0.118
  dim 4: 0.515×(-0.14) + 0.485×0.67 = -0.072 + 0.325 = 0.253

output = [0.297, 0.449, 0.118, 0.253]
```

### Step 6: Add Residual Connection

```
final_output = h + output
             = [0.5, -0.3, 0.8, 0.2] + [0.297, 0.449, 0.118, 0.253]
             = [0.797, 0.149, 0.918, 0.453]

This goes to the next layer (or is the final output).
```

### Summary of What Happened

```
┌─────────────────────────────────────────────────────────────────┐
│  INPUT: h = [0.5, -0.3, 0.8, 0.2]                              │
│                                                                 │
│  Router decided: Expert 4 (51.5%) + Expert 1 (48.5%)           │
│  Expert 2 and Expert 3: not activated (saved 50% compute!)      │
│                                                                 │
│  output = 0.515 × E₄(h) + 0.485 × E₁(h)                       │
│                                                                 │
│  OUTPUT: [0.797, 0.149, 0.918, 0.453] (after residual)          │
│                                                                 │
│  Total params in MoE layer: 4 experts worth                     │
│  Params actually used: 2 experts worth (50%)                    │
│  Knowledge available: 4 experts worth (100%)                    │
└─────────────────────────────────────────────────────────────────┘
```

## 7. The Training Challenges

MoE training is **significantly harder** than dense model training. Here are the problems and solutions:

### 7.1 Expert Collapse

#### The Problem

```
EARLY TRAINING:

  Token 1 → Router randomly picks Expert 3
  Token 2 → Router randomly picks Expert 1
  Token 3 → Router randomly picks Expert 3 (by chance, again)
  Token 4 → Router randomly picks Expert 3 (by chance, again!)

  Expert 3 has now seen 3× more data than Expert 1.
  Expert 3 becomes BETTER (lower loss on its tokens).
  Router notices Expert 3 is better → sends MORE tokens to it.

LATE TRAINING:

  Token 1 → Expert 3     (always)
  Token 2 → Expert 3     (always)
  Token 3 → Expert 3     (always)
  Token 4 → Expert 3     (always!)

  ┌──────────────────────────────────────────────────────┐
  │  Expert 1:  0% tokens    ← DEAD (never trains)      │
  │  Expert 2:  0% tokens    ← DEAD                     │
  │  Expert 3:  100% tokens  ← OVERLOADED (bottleneck)  │
  │  Expert 4:  0% tokens    ← DEAD                     │
  │                                                      │
  │  Result: 4 experts, but only 1 is used.              │
  │  Model behaves like a DENSE model.                   │
  │  All the MoE benefits are GONE. 💀                   │
  └──────────────────────────────────────────────────────┘

  This is "EXPERT COLLAPSE" — the most dangerous failure mode.
```

### 7.2 Load Balancing Loss

#### The Solution

Add an **auxiliary loss** that penalizes uneven expert usage:

```
L_total = L_language_model + α × L_balance

Where L_balance forces the router to distribute tokens evenly.
```

#### The Math (Switch Transformer formulation)

```
L_balance = α · N · Σᵢ (fᵢ · Pᵢ)

Where:
  N = number of experts
  fᵢ = fraction of tokens routed to expert i
  Pᵢ = average routing probability assigned to expert i
  α = balancing coefficient (hyperparameter, typically 0.01)
```

#### Visual: What Load Balancing Does

```
WITHOUT load balancing:

  Tokens routed to each expert (in a batch of 100 tokens):
  
  E₁: ███████████████████████████████████████████  82 tokens
  E₂: ███                                          6 tokens
  E₃: ████████                                     12 tokens
  E₄:                                              0 tokens  ← DEAD!
  
  L_balance is HIGH → big penalty → gradient pushes router to spread out

AFTER load balancing converges:

  E₁: ████████████████████████                     26 tokens
  E₂: ██████████████████████████                   27 tokens
  E₃: ███████████████████████                      23 tokens
  E₄: ████████████████████████                     24 tokens
  
  L_balance is LOW → small penalty → roughly uniform ✅
```

#### Why fᵢ × Pᵢ works mathematically

```
Imagine 4 experts and 100 tokens.
Ideal: each expert gets 25 tokens, each has probability 0.25.

  Uniform:    Σ (fᵢ · Pᵢ) = 4 × (0.25 × 0.25) = 0.25
  
  Collapsed:  f₁=1.0, P₁=1.0, others=0
              Σ (fᵢ · Pᵢ) = 1 × (1.0 × 1.0) = 1.0

  The collapsed case has 4× higher loss!
  
  Multiply by N=4:
    Uniform:   4 × 0.25 = 1.0  (minimum possible)
    Collapsed: 4 × 1.0  = 4.0  (maximum penalty)
    
  The loss is minimized when distribution is perfectly uniform.
```

### 7.3 Router Z-Loss

An additional stabilizer that prevents the router's logits from getting too large:

```
L_z = (1/B) × Σ (log Σⱼ exp(zⱼ))²

Where:
  zⱼ = raw router logit for expert j
  B = batch size

Purpose: If logits are huge (e.g., [50, -30, 45, -20]):
  → softmax becomes nearly one-hot
  → gradients vanish
  → training becomes unstable Unstable

Z-loss PENALIZES large logit magnitudes.
  → Keeps logits moderate (e.g., [2.1, -0.3, 1.7, -0.5])
  → softmax stays smooth
  → gradients flow well ✅
```

### 7.4 Token Dropping & Capacity Factor

#### The Problem: Hardware Needs Predictable Workloads

```
GPUs like UNIFORM, PREDICTABLE work. But routers create VARIABLE loads:

  Batch of 32 tokens, 4 experts, top-1 routing:

  Ideal (8 tokens each):
    E₁: ████████  8 tokens
    E₂: ████████  8 tokens
    E₃: ████████  8 tokens
    E₄: ████████  8 tokens
    
  Reality (after router):
    E₁: ██████████████  14 tokens   ← overloaded!
    E₂: ███              3 tokens   ← underused
    E₃: ██████████      10 tokens
    E₄: █████            5 tokens
    
  Problem: Expert 1 takes 2× longer than Expert 2.
  The GPU running Expert 2 sits IDLE waiting for Expert 1.
  → Wasted compute, slower training.
```

#### The Solution: Capacity Factor

```
Expert Capacity = (Total Tokens / Num Experts) × Capacity Factor

Example: 32 tokens, 4 experts, capacity factor C = 1.25
  → Capacity = (32/4) × 1.25 = 10 tokens per expert MAX

If an expert is assigned MORE than 10 tokens:
  → Overflow tokens are DROPPED (skip the expert, go via residual)

  E₁: ██████████|████  10 kept + 4 DROPPED (sent via residual)
  E₂: ███               3 tokens (all processed)
  E₃: ██████████        10 tokens (all processed)
  E₄: █████              5 tokens (all processed)
```

#### Capacity Factor Tradeoffs

```
┌─────────────────┬────────────────────────────────────────────┐
│ Capacity Factor │ Effect                                     │
├─────────────────┼────────────────────────────────────────────┤
│ C = 1.0         │ Tightest. Many tokens dropped if uneven.  │
│                 │ → Most memory efficient                    │
│                 │ → Potentially harms quality                │
├─────────────────┼────────────────────────────────────────────┤
│ C = 1.25        │ Common choice. Small buffer for overflow.  │
│                 │ → Good balance of efficiency and quality   │
├─────────────────┼────────────────────────────────────────────┤
│ C = 2.0         │ Generous. Very few tokens ever dropped.    │
│                 │ → Best quality, but wastes memory/compute  │
│                 │   on empty expert capacity buffers         │
├─────────────────┼────────────────────────────────────────────┤
│ No limit        │ No dropping at all. Fully dynamic.         │
│                 │ → Best quality but unpredictable hardware  │
│                 │   utilization (some models do this)        │
```

## 8.5 Routing Strategies — Complete Deep Dive

> **Why This Section Matters:** The router is the single most critical design decision in any MoE model. A bad routing strategy leads to expert collapse, training instability, or wasted compute. Every major MoE model innovates primarily on the routing mechanism.

### 8.5.1 Token-Choice Routing (Standard)

The default approach used in most MoE models. Each token independently selects its top-k experts.

```
TOKEN-CHOICE ROUTING:

  For each token x in the batch:
    1. Compute affinity scores:  s = x · W_router       (shape: n_experts)
    2. Apply activation:         p = softmax(s)          (probability dist)
    3. Select top-k:             experts = top_k(p, k)   (k indices)
    4. Renormalize weights:      w = softmax(p[experts])  (sum to 1.0)
    5. Output:                   y = Σ wᵢ · Expertᵢ(x)

  Example with 8 experts, top-2:

    Token "transformer" → scores: [2.1, 0.3, 3.7, -1.2, 0.8, 1.5, -0.4, 0.9]
                        → probs:  [0.09, 0.02, 0.46, 0.003, 0.03, 0.05, 0.008, 0.03]
                        → top-2:  Expert 3 (46%) + Expert 1 (9%)
                        → renorm: [0.84, 0.16]
                        → output: 0.84 × E₃(x) + 0.16 × E₁(x)
```

**Pros:** Simple, well-understood, naturally learns specialization
**Cons:** Load imbalance (popular experts get overwhelmed), requires auxiliary losses

### 8.5.2 Expert-Choice Routing (Reversed!)

Introduced by Zhou et al. (2022). Instead of tokens choosing experts, **experts choose tokens**.

```
EXPERT-CHOICE ROUTING — The Reversal:

  Standard:  TOKEN → "I want Expert 3 and Expert 7"
  Reversed:  EXPERT → "I want Token 12, Token 47, and Token 83"

  Algorithm:
    1. Compute ALL affinity scores: S = X · W_router     (batch × n_experts)
    2. For EACH expert i:
       - Look at column i of S (all tokens' affinity to expert i)
       - Select top-C tokens with highest affinity (C = capacity)
       - Process ONLY those tokens

  Example (batch=8 tokens, 4 experts, capacity=2 each):

    Affinity matrix S:
                 E₁    E₂    E₃    E₄
    Token 1:    0.9   0.1   0.3   0.2
    Token 2:    0.2   0.8   0.1   0.5
    Token 3:    0.7   0.3   0.6   0.1
    Token 4:    0.1   0.7   0.2   0.9
    Token 5:    0.6   0.2   0.8   0.3
    Token 6:    0.3   0.5   0.9   0.4
    Token 7:    0.4   0.6   0.4   0.8
    Token 8:    0.8   0.4   0.5   0.6

    Expert 1 picks: Token 1 (0.9), Token 8 (0.8)
    Expert 2 picks: Token 2 (0.8), Token 4 (0.7)
    Expert 3 picks: Token 6 (0.9), Token 5 (0.8)
    Expert 4 picks: Token 4 (0.9), Token 7 (0.8)

    Result:
    ✅ Perfect load balance — each expert has exactly 2 tokens
    ✅ Token 4 gets processed by TWO experts (variable assignment!)
    ⚠️ Token 3 got picked by ZERO experts (orphaned!)
       → Token 3 only passes through the residual connection
```

**Key Insight:** Expert-choice naturally achieves perfect load balance without
needing any auxiliary loss. However, some tokens may be ignored, and the variable
number of experts per token makes it harder to use during autoregressive inference.

**Used by:** Google's "Mixture-of-Experts with Expert Choice Routing" (2022)

### 8.5.3 Hash Routing (Deterministic)

The simplest possible routing — no learned parameters at all. Tokens are assigned
to experts by a hash function.

```
HASH ROUTING — No Learning Required:

  expert_id = hash(token_id) % num_experts

  Example (8 experts):
    "the"      → hash("the")      % 8 = 3  → Expert 3
    "quantum"  → hash("quantum")  % 8 = 7  → Expert 7
    "physics"  → hash("physics")  % 8 = 1  → Expert 1
    "is"       → hash("is")       % 8 = 5  → Expert 5

  Variants:
  ┌─────────────────────────────────────────────────────────────┐
  │ Random Hash:  Truly random assignment per token             │
  │ Position Hash: Based on token position in sequence          │
  │ Cluster Hash:  Similar tokens → same expert (via LSH)      │
  └─────────────────────────────────────────────────────────────┘
```

**Pros:** Perfect load balance (uniform hash), zero router overhead, no training instability
**Cons:** No semantic awareness — "quantum" and "cooking" could go to the same expert.
Significantly lower model quality than learned routing.
**Used as:** A baseline/ablation, rarely in production. Cerebras explored this for throughput.

### 8.5.4 BASE Layers (Linear Assignment)

BASE (Lewis et al., 2021) treats routing as an **optimal assignment problem**
solved with linear programming.

```
BASE LAYER — Optimal Token-to-Expert Matching:

  Problem: Given N tokens and E experts, find the assignment
           that MAXIMIZES total affinity while ensuring each
           expert gets exactly N/E tokens.

  This is a classic "balanced assignment problem" from operations research.

  Step 1: Compute affinity matrix A (N × E)
          Aᵢⱼ = how well token i matches expert j

  Step 2: Solve the linear assignment:
          maximize  Σᵢⱼ Aᵢⱼ · xᵢⱼ
          subject to:
            Σⱼ xᵢⱼ = 1       (each token goes to exactly 1 expert)
            Σᵢ xᵢⱼ = N/E     (each expert gets exactly N/E tokens)
            xᵢⱼ ∈ {0, 1}

  Step 3: Use the optimal assignment for routing.

  Example (6 tokens, 3 experts, capacity 2 each):

    Affinity matrix:
              E₁    E₂    E₃
    Token A:  0.9   0.3   0.1    → Best match: E₁
    Token B:  0.8   0.5   0.2    → Best match: E₁ (but E₁ is full!)
    Token C:  0.2   0.7   0.4    → Best match: E₂
    Token D:  0.1   0.6   0.8    → Best match: E₃
    Token E:  0.4   0.2   0.9    → Best match: E₃ (but E₃ is full!)
    Token F:  0.3   0.8   0.5    → Best match: E₂ (but E₂ is full!)

    Optimal assignment (via LP solver):
    E₁: Token A (0.9) + Token B (0.8)  → total: 1.7
    E₂: Token C (0.7) + Token F (0.8)  → total: 1.5
    E₃: Token D (0.8) + Token E (0.9)  → total: 1.7
    Grand total affinity: 4.9 (maximized!)
```

**Pros:** Globally optimal assignment, perfect balance, no auxiliary loss needed
**Cons:** Requires solving an optimization problem per batch (expensive),
doesn't scale to very large batches, top-1 only

### 8.5.5 Soft MoE (Fully Differentiable)

Soft MoE (Puigcerver et al., 2023 — Google) removes hard routing entirely.
ALL experts process ALL tokens, but with soft weighted inputs.

```
SOFT MoE — No Hard Decisions:

  Instead of routing tokens to experts, Soft MoE creates
  "virtual tokens" that are soft combinations of all real tokens.

  Step 1: Compute dispatch weights D (N_tokens × N_slots)
          D = softmax(X · Φ, axis=tokens)   ← per-slot normalization

  Step 2: Create virtual inputs for each expert
          X̃ = Dᵀ · X      ← each "slot" is a weighted mix of ALL tokens

  Step 3: Process virtual inputs through experts
          Ỹᵢ = Expertᵢ(X̃ᵢ)   ← each expert processes its slot

  Step 4: Compute combine weights C (N_tokens × N_slots)
          C = softmax(X · Φ, axis=slots)    ← per-token normalization

  Step 5: Reconstruct real token outputs
          Y = C · Ỹ       ← each token recombines expert outputs

  Visual:

    Real tokens:   [T₁] [T₂] [T₃] [T₄] [T₅]
                    ↓↘↘  ↓↘   ↓   ↙↓  ↙↙↓    ← soft dispatch
    Virtual:       [S₁]        [S₂]          (weighted mixes)
                    ↓            ↓
                  [Expert 1]   [Expert 2]     ← process
                    ↓            ↓
    Virtual out:   [Ỹ₁]        [Ỹ₂]
                    ↓↗↗  ↓↗   ↓   ↘↓  ↘↘↓    ← soft combine
    Real outputs:  [Y₁] [Y₂] [Y₃] [Y₄] [Y₅]
```

**Pros:** Fully differentiable, no load balancing issues, no token dropping, very stable
**Cons:** ALL experts activated for every token → higher compute. Not suitable for very
large MoE models. Cannot be used for autoregressive generation easily.
**Used by:** Explored by Google Research for vision models (ViT-based)

### 8.5.6 Sigmoid Gating (DeepSeek V3 Innovation)

DeepSeek V3 replaces softmax with **sigmoid** in the router — a subtle but powerful change.

```
SOFTMAX vs SIGMOID GATING:

  SOFTMAX (traditional):
    probs = softmax(scores) = exp(sᵢ) / Σⱼ exp(sⱼ)

    Properties:
    • All probs sum to 1.0 (forced normalization)
    • Experts COMPETE: raising one score lowers all others
    • With 256 experts, even the best expert might only get 0.02
    • Hard to distinguish "this expert is great" from "all are okay"

    Example (4 experts):
      scores:  [3.0,  2.8,  0.5,  0.1]
      softmax: [0.42, 0.34, 0.08, 0.05]  ← forced competition

  SIGMOID (DeepSeek V3):
    probs = σ(sᵢ) = 1 / (1 + exp(-sᵢ))

    Properties:
    • Each score is INDEPENDENT (no forced normalization)
    • Multiple experts can have high affinity simultaneously
    • Better signal for "this input needs physics AND math"
    • No artificial competition between experts

    Example (4 experts):
      scores:  [3.0,  2.8,  0.5,  -2.0]
      sigmoid: [0.95, 0.94, 0.62, 0.12]  ← independent scoring
      top-2 selected, then renormalized for weighted sum
```

**Why it matters:** With 256 routed experts, softmax spreads probability so thin
that distinguishing genuinely relevant experts becomes noisy. Sigmoid scores
each expert independently, giving clearer and more stable routing signals at scale.

### 8.5.7 Auxiliary-Loss-Free Load Balancing (DeepSeek V3)

Traditional MoE models add an auxiliary loss (L_balance) to force even expert usage.
But this loss **fights against** the main language modeling objective.

```
TRADITIONAL APPROACH (auxiliary loss):

  L_total = L_language_model + α × L_balance
                                 ↑
              This can hurt language modeling quality!
              High α → good balance but worse language model
              Low α  → good language model but expert collapse
              Finding the right α is painful.


DEEPSEEK V3 APPROACH (bias adjustment):

  Each expert i has a bias term bᵢ (NOT trained by gradient descent).

  Routing decision:
    gating_score = sigmoid(x · wᵢ + bᵢ)    ← bias added BEFORE selection
                                     ↑
                              adjustable knob

  After each training step:
    For each expert i:
      if expert i is OVERLOADED (got too many tokens):
        bᵢ = bᵢ - γ        ← decrease bias → fewer tokens routed here
      if expert i is UNDERLOADED (got too few tokens):
        bᵢ = bᵢ + γ        ← increase bias → more tokens routed here

  CRITICAL: The bias only affects ROUTING DECISIONS (which experts
  are selected). It does NOT affect the gating WEIGHTS used in the
  final weighted sum. So the language model quality is untouched!

  Step-by-step example:
  ┌─────────────────────────────────────────────────────────────────┐
  │ Step 1000: Expert 42 has bias = 0.0                            │
  │   → Gets 3.2% of tokens (target: 0.39% = 1/256)               │
  │   → WAY overloaded!                                            │
  │   → bias₄₂ ← 0.0 - 0.001 = -0.001                            │
  │                                                                 │
  │ Step 2000: Expert 42 has bias = -0.05                           │
  │   → Gets 0.8% of tokens                                        │
  │   → Still overloaded                                            │
  │   → bias₄₂ ← -0.05 - 0.001 = -0.051                          │
  │                                                                 │
  │ Step 5000: Expert 42 has bias = -0.12                           │
  │   → Gets 0.41% of tokens                                       │
  │   → Close to target! ✅                                         │
  │   → bias stays stable                                           │
  └─────────────────────────────────────────────────────────────────┘
```

**Result:** Load is balanced without ANY compromise to language modeling quality.
This is considered one of DeepSeek V3's key innovations.

### 8.5.8 Hierarchical Routing (DeepSeek V3)

With 256 experts, a flat router can be noisy and hard to train.
DeepSeek V3 uses a **two-stage hierarchical** approach:

```
HIERARCHICAL (2-STAGE) ROUTING:

  256 experts organized into GROUPS:

  ┌─────────┐ ┌─────────┐ ┌─────────┐     ┌─────────┐
  │ Group 1 │ │ Group 2 │ │ Group 3 │ ... │ Group 8 │
  │ E1..E32 │ │ E33..E64│ │ E65..E96│     │E225..256│
  └─────────┘ └─────────┘ └─────────┘     └─────────┘

  Stage 1 — GROUP selection:
    group_scores = x · W_group       (d_model × 8 groups)
    selected_groups = top_g(group_scores, g=2)

  Stage 2 — EXPERT selection within chosen groups:
    For each selected group:
      expert_scores = x · W_expert_group  (d_model × 32 experts)
      selected_experts = top_k(expert_scores, k=4)

  Total: 2 groups × 4 experts = 8 experts per token

  EXAMPLE:
    Token "derivative" (math context)

    Stage 1:
      Group 1 (science):      0.82  ← selected!
      Group 2 (language):      0.15
      Group 3 (code):          0.71  ← selected!
      Group 4 (creative):      0.08
      ...

    Stage 2 (within Group 1):
      E3 (calculus):           0.91  ← selected
      E17 (linear algebra):    0.74  ← selected
      E28 (physics):           0.68  ← selected
      E8 (statistics):         0.62  ← selected

    Stage 2 (within Group 3):
      E71 (numerical methods): 0.85  ← selected
      E80 (symbolic math):     0.78  ← selected
      E69 (algorithms):        0.55  ← selected
      E93 (debugging):         0.41  ← selected


  WHY HIERARCHICAL?
  ┌──────────────────────────────────────────────────────────────┐
  │ Flat routing (256):    256 scores to compute and compare     │
  │ Hierarchical (8 + 32): 8 scores + 2 × 32 = 72 scores       │
  │                                                              │
  │ Also: experts in the same group tend to be related,          │
  │ so the router learns "topic → sub-topic" structure.         │
  └──────────────────────────────────────────────────────────────┘
```

**Also enables device-limited routing:** Groups can map to physical GPUs, keeping
all-to-all communication within a limited set of nodes → massive latency reduction.

## 9. Shared Experts — The "Always-On" Innovation

### The Problem: Knowledge Redundancy

```
WITHOUT shared experts:

  Expert 1 needs to know: grammar + physics
  Expert 2 needs to know: grammar + poetry  
  Expert 3 needs to know: grammar + code
  Expert 4 needs to know: grammar + music
                           ↑
                     GRAMMAR IS DUPLICATED 4 TIMES!
                     
  Each expert wastes capacity on common knowledge
  that every token needs regardless of domain.
```

### The Solution: Isolate Common Knowledge

Introduced by DeepSeek, adopted by Gemma 4:

```
WITH shared expert:

  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │   SHARED EXPERT (always active for EVERY token):         │
  │   ┌──────────────────────────────────┐                   │
  │   │  Learns: grammar, syntax, common │                   │
  │   │  language patterns, basic logic  │  ← always ON 🔥   │
  │   └──────────────────────────────────┘                   │
  │                                                          │
  │   ROUTED EXPERTS (activated by router):                  │
  │   ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
  │   │ Expert 1 │ │ Expert 2 │ │ Expert 3 │ │ Expert 4 │  │
  │   │ physics  │ │ poetry   │ │ code     │ │ music    │  │
  │   │ ONLY     │ │ ONLY     │ │ ONLY     │ │ ONLY     │  │
  │   │          │ │          │ │          │ │          │  │
  │   │  💤      │ │  🔥     │ │  💤      │ │  💤      │  │
  │   └──────────┘ └──────────┘ └──────────┘ └──────────┘  │
  │                                                          │
  │   For token "the":                                       │
  │     output = Shared(x) + gate₂ × E₂(x)                  │
  │              ↑ always     ↑ router selects                │
  │                                                          │
  │   No more redundancy! Each routed expert focuses         │
  │   entirely on its specialty.                             │
  └──────────────────────────────────────────────────────────┘
```

### How Gemma 4 26B Uses This

```
Gemma 4 26B-A4B:
  128 routed experts
  + 1 shared expert (always active)
  Router selects 8 routed experts per token
  
  Active per token: 1 shared + 8 routed = 9 expert FFNs
  Total available:  1 shared + 128 routed = 129 expert FFNs
  Activation ratio: 9/129 ≈ 7%
```

## 12. Expert Parallelism

### The Problem: 128 Experts Don't Fit on One GPU

```
One A100 GPU: 80 GB VRAM
128 experts × 200M params × 2 bytes (fp16) = ~51 GB just for experts
+ attention layers, optimizer states, activations → doesn't fit!
```

### The Solution: Spread Experts Across GPUs

```
EXPERT PARALLELISM:

  GPU 0:  Experts 1-32     +  Attention layers (replicated)
  GPU 1:  Experts 33-64    +  Attention layers (replicated)
  GPU 2:  Experts 65-96    +  Attention layers (replicated)
  GPU 3:  Experts 97-128   +  Attention layers (replicated)
  
  ┌──────────────────────────────────────────────────────────┐
  │  For each token, the flow is:                            │
  │                                                          │
  │  1. ALL GPUs run attention in parallel (replicated)      │
  │  2. Router on each GPU determines expert assignments     │
  │  3. ALL-TO-ALL COMMUNICATION:                            │
  │     Token states are SENT to the GPUs holding their      │
  │     assigned experts                                     │
  │  4. Each GPU processes tokens using its local experts    │
  │  5. ALL-TO-ALL COMMUNICATION (reverse):                  │
  │     Expert outputs are sent BACK to the originating GPU  │
  │  6. Each GPU combines the expert outputs                 │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

### The Communication Pattern

```
        ┌───────── ALL-TO-ALL ─────────┐
        │                               │
  GPU 0: Token A needs Expert 45 ──────→ GPU 1 (has Expert 45)
  GPU 1: Token B needs Expert 10 ──────→ GPU 0 (has Expert 10)
  GPU 2: Token C needs Expert 100 ─────→ GPU 3 (has Expert 100)
  GPU 3: Token D needs Expert 70 ──────→ GPU 2 (has Expert 70)
        │                               │
        └───────── ALL-TO-ALL ─────────┘

  This all-to-all communication is the MAIN COST of expert parallelism.
  The network bandwidth between GPUs is the bottleneck, not compute.
  
  NVLink (within node):  600 GB/s → fast ✅
  InfiniBand (across nodes): 400 Gbps → can be a bottleneck ⚠️
```

## 13. MoE vs Dense — When to Use What

```
┌──────────────────────────────────────────────────────────────────────┐
│  CHOOSE DENSE WHEN:                                                  │
│                                                                      │
│  ✅ Small model (< 7B) — MoE overhead isn't worth it                │
│  ✅ Latency-critical — no all-to-all communication                  │
│  ✅ Simple deployment — fits on one GPU                              │
│  ✅ Fine-tuning — dense models fine-tune more predictably            │
│  ✅ Batch size 1 — MoE's parallelism benefits disappear              │
│                                                                      │
├──────────────────────────────────────────────────────────────────────┤
│  CHOOSE MoE WHEN:                                                    │
│                                                                      │
│  ✅ Want maximum intelligence per FLOP                               │
│  ✅ Have enough VRAM to hold all experts (even if sparse)            │
│  ✅ Serving many users (high batch sizes → good GPU utilization)     │
│  ✅ Diverse workloads (code + language + math → experts specialize)  │
│  ✅ Pre-training from scratch (MoE trains ~2-4× faster to converge) │
│  ✅ Scale beyond single-GPU capacity                                 │
│                                                                      │
├──────────────────────────────────────────────────────────────────────┤
│  THE CATCH:                                                          │
│                                                                      │
│  MoE 26B stores 26B params but activates 4B per token.               │
│  A dense 7B model stores 7B params and activates 7B.                │
│                                                                      │
│  Even though MoE uses fewer FLOPs per token,                         │
│  it needs MORE MEMORY to hold all the sleeping experts.              │
│                                                                      │
│  ┌────────────────────┬───────────────┬───────────────┐              │
│  │ Metric             │  Dense 7B     │  MoE 26B-A4B  │              │
│  ├────────────────────┼───────────────┼───────────────┤              │
│  │ FLOPs per token    │  7B           │  4B (faster!) │              │
│  │ Memory needed      │  14 GB (fp16) │  52 GB (fp16) │              │
│  │ Intelligence       │  ████████     │  ████████████ │              │
│  │ Tokens/second      │  Fast         │  Faster FLOPs │              │
│  │                    │               │  but memory-  │              │
│  │                    │               │  bandwidth    │              │
│  │                    │               │  limited      │              │
│  └────────────────────┴───────────────┴───────────────┘              │
│                                                                      │
│  MoE trades MEMORY for INTELLIGENCE.                                 │
│  Quantization (4-bit) makes the memory tradeoff manageable.          │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

## 15. Key Takeaways

```
┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  1. MoE = conditional computation.                                   │
│     Not all parameters are needed for all inputs.                    │
│     Activate only what's relevant.                                   │
│                                                                      │
│  2. The router is everything.                                        │
│     A bad router → expert collapse → model is dense.                │
│     Load balancing loss is ESSENTIAL for training.                   │
│                                                                      │
│  3. More, smaller experts > fewer, larger experts.                   │
│     Fine-grained → true specialization.                              │
│     Shared experts → no redundancy.                                  │
│                                                                      │
│  4. MoE trades memory for intelligence.                              │
│     All experts must be in memory (high VRAM).                       │
│     But only a few run per token (low FLOPs).                       │
│     Quantization makes this tradeoff practical.                      │
│                                                                      │
│  5. Expert parallelism is the deployment strategy.                   │
│     Experts spread across GPUs.                                      │
│     All-to-all communication is the bottleneck.                      │
│                                                                      │
│  6. MoE is the dominant paradigm for frontier models.                │
│     GPT-4, Gemini, DeepSeek V3, Gemma 4, Qwen2                     │
│     all use MoE. Dense-only models are becoming the exception.       │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

## Quick Reference Formula Card

```
┌──────────────────────────────────────────────────────────────────┐
│                        MoE LAYER                                 │
│                                                                  │
│  Router:  g(x) = softmax(x · W_r)         → expert probabilities│
│  Top-k:   select k experts with highest g(x)ᵢ                   │
│  Output:  y = Σᵢ∈top-k  gᵢ(x) · Eᵢ(x)   → weighted expert sum │
│                                                                  │
│  Load Balancing Loss:                                            │
│    L_bal = α · N · Σᵢ fᵢ · Pᵢ                                  │
│    fᵢ = fraction of tokens to expert i                          │
│    Pᵢ = mean routing probability to expert i                    │
│                                                                  │
│  Expert Capacity:                                                │
│    C_expert = (total_tokens / N) × capacity_factor               │
│                                                                  │
│  With Shared Expert:                                             │
│    y = E_shared(x) + Σᵢ∈top-k  gᵢ(x) · Eᵢ(x)                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```
