## Slide 2 — The Efficiency Challenge in Modern LLMs

### The Problem — Why This Matters

**The scale crisis in numbers:**
- GPT-4 class models are estimated at ~1.8 trillion parameters. To serve one such model at FP16 precision, you need roughly `1.8T × 2 bytes = 3.6 TB` of GPU memory just for the weights
- That's approximately 45 NVIDIA A100 80GB GPUs — just to *load* the model, before any computation
- LLaMA 3 405B (the largest open model before Gemma 4): needs 8× A100 80GB GPUs (~$200K+ in hardware) and costs ~$2-5 per 1M tokens to serve

**Concrete scenario — why edge deployment matters:**

Imagine you're building a health assistant app for rural Bangladesh where internet connectivity is unreliable. You need:
- The model to run *locally* on a user's phone (no server round-trip)
- A phone with 4-6 GB total RAM (of which the OS uses ~2 GB)
- So the model must fit in ~2-3 GB or less

With a standard 7B model at FP16: `7B × 2 bytes = 14 GB` — impossible.
With Gemma 4 E2B at 4-bit: `5.1B × 0.5 bytes ≈ 2.5 GB` — but only 2.3B params are active per token, so compute is even cheaper.

**The three central questions:**

1. **Cost question:** Can we get 31B-level reasoning at 10-100× lower cost? Gemma 4's answer: Yes, the E4B achieves weighted accuracy 0.675 while using only 14.9 GB VRAM (vs. 48+ GB for equivalently-capable models)

2. **Size question:** Can 2B params match 7B? This seems impossible because a 2B model has fundamentally fewer "slots" to store knowledge. Gemma 4's answer: PLE (Per-Layer Embeddings) lets 2.3B *effective* params carry representational depth equivalent to a 7B dense model, because each layer gets its own specialized embedding signal

3. **Deployment question:** Can a model run on a phone with 1.5 GB RAM? Gemma 4 E2B at 4-bit quantization: ~1.5 GB. It was literally tested on Raspberry Pi hardware

**The "20-40× compression" claim — where does it come from?**

Start from a 31B dense model as the intelligence baseline:
```
31B × 16-bit = 62 GB  (baseline)
After all 5 techniques + 4-bit quant:
E2B at 4-bit ≈ 1.5 GB
62 GB / 1.5 GB ≈ 41× compression
```
And the quality retained is ~90-95% of the 31B baseline. That's the headline result.

## Slide 3 — Gemma 4 Model Family at a Glance

### Reading the Table

**E2B and E4B — "Effective" naming explained:**

The "E" stands for "Effective" parameters. This is a new naming convention Google introduced. Here's why:

- E2B has **5.1 billion total parameters** stored on disk/memory
- But for any given input token, only **2.3 billion** are actually used in computation
- The extra ~2.8B are in the PLE (Per-Layer Embedding) tables — large lookup tables where only one row per token per layer is accessed

**Analogy:** Think of a dictionary (PLE table) vs. a calculator (the model's weights). The dictionary has 256,000 entries (one per vocabulary token), but for any given word, you only look up ONE entry. The dictionary is big (storage), but looking up one entry is cheap (compute).

**Why this distinction matters for your thesis:**
- Traditional model comparison: "Model A has 7B params, Model B has 2B params, so A is bigger/better"
- Gemma 4 breaks this: E2B has 5.1B stored but only 2.3B active. How do you compare it?
- This creates a research question: what's the right metric — total params, active params, or performance-per-active-param?

**26B A4B — the MoE model:**

- "A4B" = "Active 4 Billion" — only 3.8B of the 25.2B total params fire per token
- It has 128 expert FFNs, but only 8 + 1 shared are selected per token
- Think of it as: the model has the *knowledge* of 25.2B parameters (because all experts were trained), but the *speed* of a ~4B model (because only 9 experts run)

**Scenario — choosing the right model:**

| Scenario | Best Model | Why |
|:---------|:-----------|:----|
| Health chatbot on a phone (rural, offline) | E2B | Fits in 1.5 GB at 4-bit, handles basic Q&A |
| Code assistant on a laptop with 16 GB RAM | E4B | Stronger reasoning, fits at 4-bit (~3 GB) |
| Enterprise document analysis (128K docs) | 26B MoE | 256K context, near-31B quality at 4B cost |
| Research fine-tuning for best possible quality | 31B Dense | Full dense model, best for adaptation |

**The "same training data" insight:**

All four models share the same training pipeline — they were all distilled from the same Gemini 3 family teacher, using the same data mix. This is why Gemma 4 is a natural controlled experiment for studying compression: the ONLY variable between models is the architecture/compression strategy. If you observe a performance difference on a specific demographic or task, you can attribute it to the architecture, not the data.

**Multimodality note:**
- E2B/E4B: text + image + **audio** (native audio is rare at this size)
- 26B/31B: text + image + **video** (process sequences of frames)
- All models support a "thinking mode" for step-by-step reasoning
- All support system prompts and function calling (agentic use)

## Slide 4 — Five Interlocking Efficiency Techniques

### The Big Picture — Why Five Techniques, Not One?

The key insight to convey: **each technique targets a DIFFERENT bottleneck**, and they compose without interfering with each other. No single trick gets you 20-40× compression — you need the stack.

**The bottleneck map:**

```
An LLM's costs:
│
├── TRAINING COST ──────→ Solved by: Knowledge Distillation (KD)
│   (How well can a small model learn?)
│
├── REPRESENTATION COST ─→ Solved by: Per-Layer Embeddings (PLE)
│   (Can a narrow model represent rich semantics?)
│
├── ATTENTION COST ──────→ Solved by: Hybrid Attention + p-RoPE
│   (O(n²) attention at 128K-256K tokens)
│
├── FFN COMPUTE COST ───→ Solved by: Mixture-of-Experts (MoE)
│   (FFN is ~2/3 of a transformer's compute)
│
└── INFERENCE MEMORY ───→ Solved by: Shared KV Cache + Quantization
    (KV cache grows linearly with context)
```

**The "no interference" property:**

- PLE adds embedding tables → doesn't change attention or FFN math
- Hybrid attention changes which tokens attend to which → doesn't change the FFN
- MoE replaces the FFN with experts → doesn't change the attention mechanism
- Shared KV cache reuses inference-time memory → doesn't change any weights
- KD is a training technique → doesn't change the architecture at all

This means the compression ratios **multiply**:
`8× (KD) × 1.5× (PLE quality boost) × 6× (hybrid attention) × 6.5× (MoE) × 2× (shared KV) × 4× (quantization)`

**Scenario for your supervisor:**
"If I only had quantization, I'd get 4× compression. If I only had MoE, I'd get ~6.5× compute reduction. But Gemma 4 stacks ALL of them — and because they target different bottlenecks, they don't cancel each other out. That's why you get 20-40× total, not just 4×."

## Slide 5 — Technique 1: Per-Layer Embeddings (PLE)

### The Problem PLE Solves (with scenario)

**Standard transformer embedding — the "one notebook" problem:**

In a normal transformer, when the model sees the word "bank", it creates ONE embedding vector at the input, say: `[0.3, -0.1, 0.8, 0.5, ...]`

This single vector must serve ALL 36 layers:
- Layer 1 needs to know: "bank" is a noun (syntactic role)
- Layer 12 needs to know: "bank" could be financial or riverside (disambiguation)
- Layer 30 needs to know: in this context, "bank" is financial because of "interest rate" earlier

But all layers start with the SAME vector! Layer 30 has to reconstruct the financial meaning from the same initial signal that Layer 1 used for syntax.

**The compromise:** The embedding becomes a "jack of all trades" — decent for every layer, optimal for none. This is fine when you have a wide model (4096-dimensional embeddings have enough room). But in a narrow model (E2B with ~2048 dimensions), there isn't enough room in one vector to encode everything.

**PLE's solution — the "class-specific cheat sheet":**

```
For token "bank":

Standard:
  Layer 1 input:  main_embed("bank") = [0.3, -0.1, 0.8, 0.5, ...]
  Layer 12 input: main_embed("bank") = [0.3, -0.1, 0.8, 0.5, ...]  ← same!
  Layer 30 input: main_embed("bank") = [0.3, -0.1, 0.8, 0.5, ...]  ← same!

With PLE:
  Layer 1 input:  main_embed("bank") + PLE_L1("bank")
                  = [0.3, -0.1, 0.8, 0.5] + [0.01, 0.03, -0.02, 0.0]
                  Each layer gets a slightly DIFFERENT starting signal
  
  Layer 12 input: main_embed("bank") + PLE_L12("bank")
                  = [0.3, -0.1, 0.8, 0.5] + [0.05, -0.01, 0.04, -0.03]
                  Layer 12 gets a signal tuned for disambiguation!
  
  Layer 30 input: main_embed("bank") + PLE_L30("bank")
                  = [0.3, -0.1, 0.8, 0.5] + [-0.02, 0.06, 0.01, 0.02]
                  Layer 30 gets a signal tuned for reasoning!
```

**Why the PLE residuals are TINY:**
- Notice the PLE values are ~10-100× smaller than the main embedding
- They're not replacing the main embedding — they're *nudging* it
- This is similar to how residual connections work (h + f(h)) — the PLE is a small correction

**The "Effective" parameter counting:**

```
E2B totals:
  Main model weights:     ~2.3B params  (always active, always computed)
  PLE tables (36 layers): ~2.8B params  (stored, but per token only 36 rows accessed)

  For token "bank" at one forward pass:
    Active model weights: 2.3B
    Active PLE rows:      36 layers × 1 row × small_dim = ~36 × 64 = 2,304 values
    
    That's 2,304 PLE values vs 2,300,000,000 model params
    PLE compute overhead: essentially ZERO (< 0.001%)
    PLE storage overhead: ~2.8B extra params that sit in memory
```

**Why not just make the model bigger instead?**

| Approach | Extra Params | Extra Compute | Quality Gain |
|:---------|:------------|:-------------|:-------------|
| Double width (2048→4096) | +Billions (4× everywhere) | +4× per token | ~2× better representations |
| Add more layers (36→72) | +Billions (2× layers) | +2× per token | Diminishing returns |
| PLE (Gemma 4) | +576M (tables only) | +~0% (lookups) | ~50% of doubling benefit |

PLE gives you half the benefit at 0.001% of the compute cost. That's the efficiency insight.

## Slide 6 — Technique 2: Hybrid Attention — Local + Global

### The Quadratic Problem (with concrete numbers)

**Why O(n²) is a crisis at long context:**

The attention mechanism computes a score between EVERY pair of tokens:

```
Context = 1,000 tokens:    1,000 × 1,000 = 1 million scores      ✓ fine
Context = 10,000 tokens:   10K × 10K = 100 million scores         ⚡ getting expensive
Context = 128,000 tokens:  128K × 128K = 16.4 BILLION scores      💀 per layer!

With 36 layers:  36 × 16.4B = 590 BILLION attention scores total
```

On an A100 GPU doing ~300 TFLOPS, computing 590B scores alone takes a non-trivial fraction of the total inference budget. And this is just the attention — you also have FFN, embedding lookups, etc.

**Sliding window attention — the "whispering" trick:**

Instead of every token seeing every other token, each token only looks at its immediate neighbors:

```
Sentence: "The cat sat on the mat because it was very comfortable today"

Token "it" (position 7) with sliding window = 4:
  Can see: positions 3, 4, 5, 6, [7], 8, 9, 10, 11
  Cannot see: positions 0, 1, 2
  
  Cost: O(window_size) = O(1024) regardless of sequence length!
  
  At 128K context with window=1024:
    Each token: 1024 attention scores (not 128,000)
    Savings: 128,000 / 1,024 = 125× fewer scores per token
```

**But wait — doesn't this lose information?**

Not if you stack multiple local layers! Information cascades:

```
Layer 1 (local, window=1024): token at pos 50,000 knows about pos 49,000-51,000
Layer 2 (local): token at pos 50,000 knows about pos 48,000-52,000 (indirect)
Layer 3 (local): range expands to ~47,000-53,000
Layer 4 (local): ~46,000-54,000
Layer 5 (local): ~45,000-55,000

After 5 local layers: each token has INDIRECT access to ~5,000 tokens of context

Layer 6 (GLOBAL): every token directly sees ALL 128,000 tokens
→ Now the model confirms and corrects any indirect information
→ "it" directly resolves to "cat" even if they're 50,000 positions apart
```

**The party analogy (for presenting):**

"Imagine 36 people at a party. In a normal transformer, everyone is shouting at everyone simultaneously — chaos, expensive. Gemma 4's approach: 5 rounds of whispering to your neighbors, then 1 round where someone grabs a microphone and broadcasts to the whole room. After just one broadcast, everyone is synchronized. The next 5 whisper rounds refine the details."

**p-RoPE — why it matters for long context:**

Standard RoPE at position 200,000:
```
Each dimension pair gets rotated by: angle = position × θᵢ
At position 200,000 with θ₁ = 0.001:  rotation = 200 radians = ~31.8 full turns
→ The "semantic meaning" encoded in this dimension gets rotated 31 times
→ Accumulated floating-point errors corrupt the signal
```

p-RoPE (p=0.25):
```
Only 25% of dimension pairs (16 of 64) get rotated
The other 75% (48 pairs) are NEVER rotated — they encode pure meaning
→ At position 200,000, the meaning signal is PERFECTLY preserved in 48 pairs
→ Only 16 pairs carry position info (and they can handle it with proper base frequency)
```

**Why apply p-RoPE only to global layers:**
- Local layers: max position is 1024 (the window size) — standard RoPE handles 1024 positions perfectly
- Global layers: max position is 256,000 — standard RoPE would accumulate 250× more rotation error
- So you only pay the "cost" of p-RoPE (reduced positional precision from fewer rotated dims) where you actually need it

**The combined memory savings math:**

```
Setup: 128K tokens, 36 layers, each full KV cache = 1 "unit"

All-global baseline:       36 layers × 1 full unit = 36 units

Hybrid 5:1 (Gemma 4):
  6 global layers:         6 × 1 full unit = 6 units
  30 local layers:         30 × (1024/128,000) = 30 × 0.008 = 0.24 units
  Total:                   6.24 units

Savings: (36 - 6.24) / 36 = 82.7% ≈ 83% less KV cache memory

And this is BEFORE shared KV cache (Slide 7) provides another 44% on top!
```

## Slide 7 — Technique 3: Shared KV Cache

### What is the KV Cache? (Foundation)

During autoregressive generation (when the model generates one token at a time), it needs to "remember" all previous tokens. Specifically, it stores the **Key** and **Value** vectors computed for every previous token at every layer. This is the KV cache.

**Why it exists:**
```
Generating token #5001 in a conversation:

Without KV cache:
  Recompute attention for ALL 5000 previous tokens at ALL 36 layers
  → 5000 × 36 = 180,000 attention computations. Incredibly wasteful.

With KV cache:
  Store K, V from tokens 1-5000 (already computed)
  Only compute Q for the NEW token #5001
  Compute attention: Q_new × K_cached → scores → weighted V_cached
  → Only 36 new computations (one per layer)
```

**The memory explosion problem:**
```
For Llama-2-70B at 128K context in FP16:
  Per layer:   2 × 128,000 × 8 heads × 128 dim × 2 bytes = ~524 MB
  All 80 layers: 80 × 524 MB ≈ 41 GB just for KV cache!
  
  That's 41 GB ON TOP of the ~140 GB for the model weights.
  Total: ~181 GB. Even 3× A100 80GB GPUs aren't enough.
```

### How Gemma 4 Solves This

**The observation:** Researchers measured the cosine similarity between KV representations at consecutive layers and found that deep layers produce nearly identical K,V vectors.

**Intuition:** 
- Layer 1 and Layer 2 are doing fundamentally different things (syntax vs. basic semantics) → their KV representations are very different
- Layer 34 and Layer 35 are both doing high-level reasoning refinement → their KV representations are almost the same
- Recomputing nearly-identical KV at Layer 35 when Layer 34's KV already exists is pure waste

**The detective analogy:**
"Imagine 36 detectives investigating a case. Detectives 1-20 each find genuinely different clues — their notes are unique and valuable. But detectives 21-36 are all looking at the same final evidence and writing nearly identical reports. Shared KV Cache says: 'Detectives 21-36, just use Detective 20's notes instead of rewriting them.'"

**Important nuance — sharing is per attention TYPE:**
```
Layer 18 (local sliding window):  computes own KV   ← last unique local
Layer 19 (global full attention): computes own KV   ← last unique global
Layer 20 (local):                 shares L18's KV
Layer 21 (global):                shares L19's KV
Layer 22 (local):                 shares L18's KV
...

Local layers share with the LAST unique local layer's KV
Global layers share with the LAST unique global layer's KV
They don't mix — because local and global attention have different receptive fields
```

**The savings math:**
```
Without sharing: 36 unique KV caches = 36 units
With sharing:    20 unique + 16 shared = 20 units
Savings: (36-20)/36 = 44%

Combined with hybrid attention (83%):
  From baseline 36 units → 6.24 units (hybrid) → ~3.5 units (hybrid + shared)
  Total savings: ~90% less KV cache memory vs. a standard transformer!
```

## Slide 8 — Technique 4: Mixture-of-Experts (MoE)

### The Core Efficiency Insight

In a standard dense transformer, the FFN (feed-forward network) takes up roughly 2/3 of total parameters and compute. For a 31B model, that's ~20B parameters in FFN layers, ALL activated for EVERY single token.

**The waste:**
```
Token "The" (a common article):
  Dense model: runs through 20B FFN params
  Does "The" really need 20B params of processing? NO.
  Most of those params are specialized for code, science, math, etc.
  For "The", a simple grammar pattern suffices.

Token "∂f/∂x" (partial derivative):
  Dense model: same 20B FFN params
  Now these specialized math params are useful!
  But the grammar, poetry, and cooking params are wasted.
```

**MoE's solution:** Split the FFN into 128 small specialists. For each token, a router picks the 8 most relevant + 1 shared (always-on):

```
Token "The" → Router selects: [grammar_expert, function_word_expert, ...]
Token "∂f/∂x" → Router selects: [math_expert, notation_expert, ...]

Each expert: ~30M params (vs. 20B total FFN)
Active per token: 9 × 30M = ~270M (not 20B!)
```

### The Router — How Selection Works

The router is a simple linear layer that scores all 128 experts:

```
Router input: hidden_state after attention (e.g., 2048-dim vector)
Router weights: 2048 × 128 matrix
Router output: 128 scores (one per expert)

Selection: top-8 scores win → those 8 experts + 1 shared process the token
Weighting: outputs are combined as weighted sum using softmax of selected scores

Expert_3 output × 0.23 + Expert_47 output × 0.18 + ... = final FFN output
```

**Scenario — how experts specialize (simplified):**

Over the course of training, the router learns patterns like:
- Tokens appearing in code contexts → routed to experts that learned code patterns
- Tokens in medical text → routed to experts that learned biomedical vocabulary
- Tokens in casual chat → routed to experts that learned colloquial patterns

This specialization emerges naturally from training — no one manually assigns experts to domains. The router and experts co-evolve.

**The shared expert:** One expert is ALWAYS activated (the "+1"). It handles patterns common to ALL tokens — basic syntax, common word relationships, etc. This ensures a baseline quality floor even if the router makes a suboptimal selection.

### Why 128 Experts and Not 8 or 1024?

```
Fewer experts (8):    Coarse specialization, each expert is large (~2.5B)
                      Router has fewer choices → less efficiency gain
                      
More experts (1024):  Very fine-grained specialization, tiny experts (~2.5M)
                      But: more experts = harder to train (routing instability)
                      More all-to-all communication overhead in distributed training

128 experts:          Sweet spot — each expert (~30M) is large enough to learn
                      meaningful patterns but small enough for efficiency
                      8/128 = 6.25% activation → ~16× compute reduction in FFN
```

### The Numbers for Your Presentation

```
Total parameters:    25.2B (stored in memory — all experts)
Active per token:    ~3.8B (only 9 experts + attention + embedding)
FFN compute saved:   9/128 = 7% of FFN activated → 93% compute reduction in FFN
Overall compute:     ≈ a 4B dense model (because attention is separate)
Overall quality:     ≈ 26B dense model (because all experts were trained)

Memory tradeoff:     Must store all 25.2B params in memory
                     → mitigate with quantization (25.2B × 4-bit = ~12.6 GB)
```

## Slide 9 — Technique 5: Knowledge Distillation (GKD)

### Why Small Models Need Teachers

**The fundamental capacity problem:**

A 2B model has 2 billion "slots" to store learned patterns. A 31B model has 31 billion. Even with perfect architecture, the 2B model simply cannot discover all the patterns in the training data that the 31B model can — it will miss rare patterns, subtle relationships, and nuanced distinctions.

**Scenario — learning French capitals:**

```
Training data contains: "The capital of France is Paris" (appears 10,000 times)
                        "Lyon is the third-largest city in France" (appears 50 times)
                        "Marseille is France's second city" (appears 30 times)

31B model (lots of capacity): 
  Learns: Paris is the capital, Lyon is 3rd, Marseille is 2nd
  Also learns: the relationship between "capital" and "largest city"
  Also learns: French geography patterns in general

2B model (limited capacity):
  Learns: Paris is the capital (high frequency → definitely learned)
  Maybe learns: Lyon connection (low frequency → might be dropped)
  Probably misses: the geographic relationship pattern (too subtle)
```

### What "Dark Knowledge" Actually Is

**Hard labels (standard training):**
```
Input:  "The capital of France is ___"
Label:  "Paris" → one-hot vector [0, 0, ..., 1, ..., 0]

The model learns: "The answer is Paris." Period. Nothing else.
```

**Soft labels (from teacher):**
```
Input:  "The capital of France is ___"
Teacher output (with temperature=5):
  Paris:    0.42  ← most likely
  Lyon:     0.19  ← also French, also a city, plausible
  France:   0.15  ← word from the question, somewhat likely
  Berlin:   0.12  ← capital of another European country
  London:   0.08  ← capital of another European country
  "the":    0.001 ← grammatically impossible here

One training example now teaches:
  ✓ Paris is the answer (primary fact)
  ✓ Lyon is related to France and cities (geographic knowledge)
  ✓ Berlin and London are also capitals (category knowledge)  
  ✓ "the" doesn't work here (grammar knowledge)
  ✓ Paris is ~2× more likely than Lyon (confidence calibration)
```

This is 5× more learning per training example! The teacher's soft distribution encodes **relationships between options** — that's the "dark knowledge" that hard labels completely miss.

### Temperature — The Control Knob

```
Raw teacher logits: [15.2, 8.1, 5.3, 3.2, 2.8]

Temperature = 1 (standard softmax):
  [0.95, 0.02, 0.01, 0.01, 0.01]
  Almost all weight on Paris — student basically sees a hard label
  Dark knowledge is hidden by the sharp peak

Temperature = 5 (Gemma-style):
  [0.42, 0.19, 0.15, 0.12, 0.12]
  Smooth! Student can see: "Lyon is half as likely as Paris"
  Rich signal for learning relationships

Temperature = 20 (too flat):
  [0.22, 0.21, 0.20, 0.19, 0.18]
  Everything looks equally likely — no useful signal at all
```

### GKD — What Makes Gemma 4's Approach Special

**The train-inference mismatch problem:**

```
Standard KD:
  Training: Student reads TEACHER-generated or dataset text
  Inference: Student reads ITS OWN generated text
  
  Problem: The student has never seen its own writing style!
  It encounters word patterns and error patterns at inference
  that it never practiced during training.
  
  Like studying for an exam using someone else's notes,
  then taking the exam in your own words.
```

**GKD (Generalized Knowledge Distillation) solves this:**

```
Step 1: Student generates its own output
        "The capital of France is located in Paris"
        (notice: "located in" is a student error/quirk)

Step 2: Teacher evaluates the student's output token-by-token
        Teacher says: "At position 7, after 'is', the best continuation 
        would be 'Paris' (0.85), not 'located' (0.03)"

Step 3: Student learns from teacher's correction of ITS OWN output
        → Student learns: "When I write 'The capital of X is', 
           I should go straight to the answer, not add filler"

This is like having a tutor who watches YOU solve problems
and corrects YOUR specific mistakes, rather than just
showing you pre-solved examples.
```

**Why GKD produces better small models:**
- Standard KD: student memorizes teacher's output patterns but can't handle its own error states
- GKD: student explicitly practices generating, gets corrected on its own errors, and learns self-correction patterns
- Result: more robust at inference time, especially on open-ended generation tasks

## Slide 10 — The Compression Multiplier Stack

### How to Present This Slide

This is the "putting it all together" slide. Walk through the table row by row, emphasizing that each technique is independent:

**Row-by-row narrative:**

1. **Knowledge Distillation (~8×):** "We start with a 31B-quality teacher and train a ~4B student. The student retains ~90% of the teacher's capability. This is the biggest single compression step."

2. **PLE ("negative" compression):** "PLE doesn't compress — it actually ADDS quality. For ~15% more stored parameters, we get ~50% of the benefit of doubling the model. It's like getting a free upgrade."

3. **Hybrid Attention (6× memory):** "By using 5 local attention layers for every 1 global layer, we cut attention memory by 83%. This is what enables 128K-256K context on a phone."

4. **MoE (6.5× compute):** "The FFN — which is 2/3 of the model's compute — now only activates 7% of its parameters per token. The compute cost drops from 26B to ~4B."

5. **Shared KV Cache (2× memory):** "Deep layers reuse earlier layers' KV cache, saving another 44% on inference memory. Combined with hybrid attention, that's ~90% total KV savings."

6. **4-bit Quantization (4× storage):** "Finally, we shrink each weight from 16 bits to 4 bits. This is the easiest step — and Gemma 4 was designed from day one to quantize well."

**The key point:** "These ratios MULTIPLY because each targets a different bottleneck. That's how you get 20-40× total compression while retaining 90-95% quality. No single technique can do this alone."

**Scenario — what this means in practice:**

```
Starting point: 31B dense model at FP16
  Memory: 62 GB, Compute: 31B ops/token, KV cache: 36 full units
  Hardware needed: 1× A100 80GB GPU minimum

After full Gemma 4 compression stack (E2B at 4-bit):
  Memory: ~1.5 GB, Compute: 2.3B ops/token, KV cache: ~3.5 units
  Hardware needed: a smartphone with 4GB RAM

Same intelligence, 41× less resource usage.
```

## Slide 13 — Quantization-Friendly by Design

### Why Architecture Choices Matter for Quantization

**RMSNorm vs LayerNorm:**
```
LayerNorm:  output = (x - mean) / sqrt(var + ε) × γ + β
  - Requires computing mean AND variance
  - Subtraction can create negative values near zero
  - At low precision: small values near zero get rounded to zero → instability

RMSNorm:    output = x / sqrt(mean(x²) + ε) × γ
  - Only computes root-mean-square (no subtraction)
  - No shift parameter β → fewer values near zero
  - More numerically stable at 4-bit and 2-bit precision
```

**SwiGLU activation:**
```
Standard: ReLU(x) = max(0, x)   → sharp corner at x=0 → quantization artifacts
Swish:    x × σ(x)              → smooth curve → gradual transitions
SwiGLU:   Swish(xW₁) ⊙ (xW₂)   → smooth AND gated → robust to rounding

When you round weights from FP16 to INT4:
  ReLU: small perturbation near x=0 can flip output from 0 to positive
  SwiGLU: small perturbation causes proportionally small output change
```

**The memory footprint comparison — key numbers:**
```
Model                  FP16 Size    4-bit Size    Hardware Needed
GPT-4 class (~1.8T)   ~3,600 GB    ~900 GB       Datacenter (45+ A100s)
LLaMA 3 70B           ~140 GB      ~35 GB        4× A100 or 1× H100
Gemma 4 31B           ~62 GB       ~16 GB        1× RTX 4090 or A100
Gemma 4 26B MoE       ~50 GB       ~12.6 GB      1× RTX 4090
Gemma 4 E4B           ~16 GB       ~3 GB         Smartphone (8GB RAM)
Gemma 4 E2B           ~10 GB       ~1.5 GB       Smartphone (4GB RAM)
```

**Expert-specific quantization in MoE — a fairness angle:**
"In the 26B MoE, not all experts are equally important. Some activate frequently (handling common English patterns), others activate rarely (handling niche domains). Mixed-precision quantization assigns 4-bit to frequent experts and 2-bit to rare ones. This raises a research question: do 'rare' experts disproportionately serve underrepresented groups? If so, giving them lower precision could amplify bias."

## Slide 14 — Key Takeaways

### Expanded Notes for Each Takeaway

**1. Architecture-First Compression:**
"Traditional compression workflow: train a large model → apply pruning/quantization/distillation after training. Gemma 4 inverts this: design the architecture WITH compression in mind from the start. PLE, hybrid attention, and MoE aren't applied after training — they ARE the architecture. This means the model never needs to 'unlearn' being large."

**Why this matters for your thesis:** This approach is philosophically different from the pruning/quantization papers in your roadmap (SparseGPT, GPTQ, AWQ). Those are post-hoc; Gemma 4 is pre-hoc. Your thesis could explore: does architecture-first compression preserve fairness better than post-hoc compression?

**2. Complementary Technique Stack:**
"The five techniques target five different bottlenecks and don't interfere with each other. This is why they compose multiplicatively. If PLE conflicted with MoE (e.g., PLE tables couldn't work inside expert FFNs), the gains wouldn't stack. The clean separation is the engineering innovation."

**3. The 'Effective Parameters' Paradigm:**
"This challenges how we think about model size. A paper that compares 'Model A (7B) vs Model B (2B)' is incomplete if Model B is an E2B-style model with PLE. The research community needs new metrics: effective params, active params per token, or performance-per-FLOP."

**4. Hardware-Aware Design:**
"Gemma 4 E2B was literally tested on Raspberry Pi. The architecture choices (RMSNorm, SwiGLU, standard FFN, clean attention) were made because they map well to existing inference engines (llama.cpp, vLLM, MLX, LiteRT). A beautiful architecture that doesn't run on any hardware is useless."
